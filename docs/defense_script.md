# 答辩讲稿与 FAQ

## 1. 开场讲稿

可以按下面这段直接讲：

> 旧版本的推荐系统只是在业务库里保存商品 embedding，然后在 Python 层做余弦相似度排序，所以它只能算推荐雏形。升级后，我把向量检索迁移到了独立向量数据库 Qdrant，商品索引同时包含 dense 语义向量、sparse 关键词向量和 ColBERT multivector。搜索侧采用 dense+sparse 混合召回、RRF 融合和 ColBERT 重排；推荐侧采用内容召回、稀疏兴趣召回、相似商品召回、协同过滤召回、热门召回、新品召回和冷启动召回，再经过统一排序、多样性控制和推荐理由生成。所以当前系统已经不是单一余弦排序，而是完整的多阶段推荐系统。  
> 同时，我还把这些能力做成了可答辩的可视化：前台能看到推荐来源和推荐理由，后台能看到 Qdrant 索引状态、推荐效果指标、调试明细和实验方案。这样老师不需要只看代码或接口 JSON，就能直接看到系统确实已经升级完成。

## 2. 演示顺序

推荐按下面顺序演示：

1. 打开后台首页。
   展示推荐 KPI、Qdrant 状态、实验方案概览。
2. 打开后台重建页。
   说明 Qdrant 连接、collection、已索引商品数和失败商品数。
3. 打开后台推荐调试页。
   输入 `user_id` 或邮箱，展示召回通道、排序特征和最终理由。
4. 打开前台首页。
   展示“猜你喜欢”的来源标签和推荐理由。
5. 打开搜索页。
   展示“语义相关 / 关键词命中 / 文化标签匹配”。
6. 打开商品详情页、购物车页、订单完成页。
   展示相似商品、购物车推荐、下单完成推荐。

## 3. 一句话概括每个模块

### 3.1 向量数据库

* PostgreSQL 存业务事实，Qdrant 存检索索引。

### 3.2 搜索系统

* dense + sparse 混合召回，RRF 融合，ColBERT 精排。

### 3.3 推荐系统

* 多路召回、统一排序、多样性控制、冷启动和解释生成。

### 3.4 后台系统

* 能看状态、能看指标、能看证据、能看实验方案。

## 4. 旧版本问题与升级后解决方式

| 旧版本问题 | 升级后如何解决 |
| --- | --- |
| 向量存在业务库 JSON 字段里 | 商品索引迁到 Qdrant collection |
| 只有 cosine 排序 | 搜索走 hybrid + rerank，推荐走多路召回 + ranker |
| 没有正式协同过滤 | 引入 sparse user index 和 item cooccurrence |
| 冷启动只是热门兜底 | 现在是正式 `cold_start` 通道 |
| 推荐结果无法解释 | 前台显示来源和理由，后台显示召回与排序证据 |
| 只有接口，没有展示 | 已有前台展示位与后台看板 |

## 5. 老师可能追问的问题

### 5.1 “你这个是不是只算余弦相似度？”

建议回答：

> 不是。余弦相似度只保留在 baseline 作为对照组。当前搜索已经是 dense+sparse 混合召回，再用 RRF 融合和 ColBERT rerank；推荐已经是内容召回、稀疏兴趣召回、相似商品召回、协同过滤、热门、新品、冷启动等多路召回，再经过 ranker 和业务重排，所以系统不是单一相似度排序。

### 5.2 “为什么一定要用向量数据库？”

建议回答：

> 因为当前系统不只是要保存 embedding，而是要做 ANN 检索、payload filter、dense/sparse/multivector 混合检索和 rerank 候选管理。如果继续把事务库当主检索库，搜索和推荐都只能停留在小规模全表扫描。Qdrant 让检索层职责独立出来，后台还能直接展示索引状态和运行时切换，这比把 embedding 放在业务库里更完整。

### 5.3 “为什么不用 pgvector，为什么选 Qdrant？”

建议回答：

> pgvector 更适合把向量能力附着在 PostgreSQL 上，而我现在需要的是独立检索层，并且要同时处理 dense、sparse 和 ColBERT multivector。Qdrant 对 named vectors、payload filter 和 hybrid retrieval 的支持更贴合当前项目，所以 PostgreSQL 负责业务事实，Qdrant 负责向量检索，这是更清晰的职责划分。

### 5.4 “协同过滤在哪里体现？”

建议回答：

> 推荐链路里已经有两条协同过滤通道：`collaborative_user` 和 `item_cooccurrence`。后台调试页可以直接看到一条推荐是否来自相似用户、是否来自商品共现关系，而不是只靠内容相似。

### 5.5 “推荐系统完整性体现在哪？”

建议回答：

> 完整性主要体现在五点：有独立向量数据库；有多路召回；有正式排序层；有冷启动和多样性；有前台解释和后台调试看板。也就是说，这不是单一算法 demo，而是完整的工程化推荐系统。

### 5.6 “怎么证明推荐更好？”

建议回答：

> 我保留了旧版 Python 余弦推荐作为 baseline，并实现了离线评估脚本，对比 `baseline`、`dense_sparse_colbert`、`multi_recall_weighted`、`multi_recall_ltr` 等多种模式。指标不仅看 `Precision@K`、`Recall@K`、`NDCG@K`，还看 `Coverage`、`Diversity` 和延迟。当前报告里 `multi_recall_ltr` 的 `P@5=0.6`、`R@5=1.0`、`NDCG@5=1.0`，明显优于 baseline，所以我不是只说“感觉更好”，而是有正式评估对照。

### 5.7 “如何解决冷启动？”

建议回答：

> 冷启动现在不是异常兜底，而是正式策略。新用户会走 `cold_start` 通道，把热门趋势、新品探索、节令主题和后台精选混排；新商品即使没有行为数据，也能通过内容向量、文化标签和探索位进入候选集合；等行为积累后，再逐步让协同过滤和排序模型接管。这说明冷启动已经是推荐链路的一部分，而不是一个临时 if-else。

## 6. 如果老师追问性能

建议回答：

> 我做了 100、1000 和 10000 商品规模的压测。结果说明 Qdrant hybrid search 可以稳定工作，但 `recommend_home` 和 `related_products` 在 10k 规模下仍然是瓶颈，这也是我在报告里明确标出的下一阶段优化方向。换句话说，我不仅做了功能，也做了性能边界证明。

## 7. 如果老师追问为什么前台要展示推荐理由

建议回答：

> 因为推荐系统答辩不能只看后台 JSON。前台把“个性化、相似商品、热门、新品探索、节令主题”等来源标签和理由直接展示出来，能证明这套系统已经进入用户可见层，而不是停留在接口或论文描述阶段。

## 8. 结束语

可以这样收尾：

> 所以这次升级的核心不是把 embedding 模型换掉，而是把系统从“余弦相似度 demo”升级成“独立向量数据库 + hybrid search + 多路召回 + 排序重排 + 前后台可视化”的完整推荐系统。这也是我对老师之前两点批评的正面回应：现在它既不是简单系统，也不是没有真正向量数据库。
