# 拾遗阁——基于向量数据库的古风文化商品智能推荐购物网站

## 项目设计文档

## 1. 项目概述

### 1.1 项目名称

**拾遗阁——基于向量数据库的古风文化商品智能推荐购物网站**

### 1.2 项目定位

“拾遗阁”是一个面向传统文化爱好者、汉服消费者、文创消费群体与非遗兴趣用户的古风文化商品智能推荐购物网站。项目以“**文化内容 + 商品电商 + 语义检索 + 个性化推荐**”为核心，打造一个不仅能“买到商品”，还能“理解商品背后文化”的 Web 应用平台。

本项目不只是一个普通商城，而是一个**融合中华优秀传统文化传播与电商服务的智能化平台**。系统通过向量数据库与语义检索技术，将用户自然语言需求、商品属性、文化知识内容进行统一建模，从而实现更符合用户兴趣与文化偏好的推荐结果。

### 1.3 项目目标

本项目拟完成以下目标：

1. 构建一个完整可用的古风文化商品购物网站，支持商品浏览、搜索、购物车、下单、会员、评价等电商基础流程。
2. 建立商品与文化知识的双重数据体系，使商品不只是“卖货”，还承担“文化讲解”的功能。
3. 引入向量数据库，支持自然语言搜索、语义相似商品召回、文化偏好匹配推荐。
4. 构建用户画像与行为分析机制，实现“猜你喜欢”“相关文化推荐”“节令专题推荐”等智能推荐能力。
5. 提供管理后台，支持商品管理、订单管理、文化内容管理、推荐数据更新与运营分析。
6. 完成可公网访问的部署版本，满足比赛展示、答辩演示与实际运行要求。该方向与今年 Web 应用小类的 B/S 形态要求是一致的。 ([中国大学生计算机设计大赛][1])

## 2. 立项背景与问题分析

### 2.1 项目背景

近年来，汉服、文创、非遗手工艺等传统文化消费快速升温，但现有电商平台大多以通用搜索和价格导向推荐为主，对传统文化商品存在以下不足：

* 商品信息呈现偏“货架式”，文化背景弱；
* 搜索依赖关键词匹配，无法理解“想要适合春日出游的宋制清雅汉服”这类语义化表达；
* 推荐逻辑偏热门商品，难以体现用户的文化兴趣偏好；
* 平台与文化传播脱节，用户无法在购物过程中获得知识增益。

因此，“拾遗阁”希望解决的不仅是“卖什么”，更是“为什么推荐给你”“它背后代表什么文化”。

### 2.2 核心痛点

本项目主要解决以下问题：

**（1）传统关键词搜索能力弱**
用户往往并不知道商品准确名称，只会表达模糊需求，如“送长辈的雅致国风礼物”“适合端午节佩戴的香囊”“偏明制、配色素净的汉服”。普通 LIKE 搜索难以命中。

**（2）商品与文化内容割裂**
用户在浏览商品时，很难获得其背后的历史、工艺、礼制、节令、地域文化等信息，平台文化传播能力不足。

**（3）推荐结果缺乏个性化与场景感**
同类平台推荐往往仅依据销量、点击量或协同过滤，缺少“节日”“场景”“文化兴趣”“审美偏好”等维度。

**（4）比赛作品常见问题是功能堆砌但后端闭环不强**
很多作品前端展示漂亮，但数据设计、业务链路、可演示性、可部署性、后端稳定性不够强。本项目将以后端闭环和推荐引擎为核心进行设计。

## 3. 比赛适配思路

本项目按软件应用与开发中的 Web 应用与开发方向设计，采用标准的浏览器/服务器模式，强调“**真实可访问、功能闭环完整、后端逻辑清晰、推荐算法可解释、文化价值鲜明**”。官方公开信息显示，该小类要求作品为 B/S 模式，并提供可通过互联网访问的网址；同时软件应用与开发类还需要按当年模板准备作品提交材料。 ([中国大学生计算机设计大赛][1])

因此，本项目在设计上明确遵循以下原则：

1. **不是 PPT 型项目，而是真实运行型项目**；
2. **不是纯静态展示，而是有数据库、有接口、有业务闭环的系统**；
3. **不是单纯商城，而是“文化电商 + 智能推荐”的完整应用**；
4. **不是只讲 AI 概念，而是把向量检索落实到可解释的产品能力中**。

## 4. 现有基础与建设原则

### 4.1 现有基础

你当前仓库已经具备较完整的前端页面雏形，包括首页、分类页、商品详情页、购物车、结算页、登录注册、个人中心、会员中心等页面，整体主题风格明确，适合作为比赛展示层基础。仓库当前公开可见文件结构主要是静态前端资源，说明下一阶段最应聚焦的是后端接口、数据库、推荐检索、管理后台与部署方案。 ([GitHub][2])

### 4.2 建设原则

为了确保项目最终能顺利落地并适合比赛答辩，本项目采用以下建设原则：

* **单体优先，模块化设计**：先做“模块化单体应用”，而不是微服务。比赛更看重稳定、可演示、可复现。
* **后端先稳，再做算法增强**：先把用户、商品、购物车、订单、会员等业务闭环打通，再加推荐系统。
* **可解释优先于花哨模型**：推荐结果必须能说明“为什么推荐”。
* **文化内容与商品内容双向关联**：推荐不仅来自商品属性，也来自文化知识文本。
* **部署可控、维护简单**：比赛版本优先考虑 Docker Compose 一键部署。

* 管理员登录

## 7. 推荐技术栈（最终建议方案）

对于你的比赛目标和当前前端基础，我建议采用**“模块化单体 + REST API + PostgreSQL/pgvector + Redis + MinIO + Docker Compose”** 这一套方案。这一套不是最潮的，但对比赛来说是**最稳、最容易讲清、最容易落地、最容易部署**的。

FastAPI 原生支持基于 OpenAPI 的自动接口文档；pgvector 提供 PostgreSQL 内的向量相似度检索；MinIO 提供 S3 兼容对象存储；Redis 既适合缓存也能承担异步消息/任务协调。这些能力很适合你这种“商品、电商业务、文化内容、向量检索、后台管理”混合型项目。 ([FastAPI][3])

### 7.1 技术栈总览

| 层级         | 推荐技术                                           | 选择理由                     |
| ---------- | ---------------------------------------------- | ------------------------ |
| 前端展示层      | 现有 HTML/CSS/JS 页面先保留，后续逐步迁移 Vue 3 + TypeScript | 兼顾你现有基础与后续扩展性            |
| Web 服务层    | Nginx                                          | 静态资源托管、反向代理、HTTPS        |
| 后端框架       | FastAPI                                        | 开发快、结构清晰、接口文档自动生成、适合比赛演示 |
| ORM / 数据建模 | SQLAlchemy 2 + Pydantic 2                      | 类型清晰、维护性好                |
| 关系数据库      | PostgreSQL                                     | 稳定、事务强、适合订单类业务           |
| 向量检索       | pgvector                                       | 与商品/订单/用户数据同库管理，部署简单     |
| 缓存 / 队列    | Redis                                          | 会话、验证码、热点缓存、异步任务协调       |
| 异步任务       | Celery 或 RQ                                    | 处理向量化、索引重建、日志统计等后台任务     |
| 对象存储       | MinIO                                          | 商品图片、文化素材、评价图片统一管理       |
| 鉴权         | JWT + Refresh Token                            | 前后端分离常用方案                |
| 搜索增强       | PostgreSQL 全文检索 + 向量召回混合                       | 实现关键词与语义的双通道检索           |
| 容器化        | Docker + Docker Compose                        | 比赛环境最容易交付和复现             |
| 接口测试       | Apifox / Postman                               | 便于联调和答辩展示                |
| 自动化测试      | Pytest                                         | 回归测试、接口测试方便              |
| 日志监控       | Loguru + Prometheus/Grafana（可选）                | 有余力再加，非首要                |

### 7.2 为什么不建议一开始上微服务

对于比赛型项目，微服务会显著增加：

* 服务拆分成本
* 接口治理成本
* 部署复杂度
* 答辩解释成本
* 排障难度

所以本项目最佳架构不是“复杂”，而是“**边界清晰的单体应用**”。后端内部按模块拆分即可：

* user 模块
* product 模块
* order 模块
* member 模块
* content 模块
* recommendation 模块
* admin 模块

## 8. 总体架构设计

### 8.1 架构风格

本系统采用**前后端分离 + 模块化单体后端 + 向量检索增强**的总体架构。

### 8.2 总体架构图（文字版）

```text
浏览器 / 前端页面
        |
        v
      Nginx
   /         \
静态页面      /api
             |
             v
          FastAPI
   ┌────────┼──────────────┬─────────────┬─────────────┐
   v        v              v             v             v
用户模块  商品模块      订单模块      会员模块      内容模块
                                      |
                                      v
                                  推荐模块
   ┌──────────────┬──────────────┬──────────────┐
   v              v              v
PostgreSQL     Redis          MinIO
(关系数据+     (缓存/任务)     (图片/素材)
 pgvector)
             |
             v
         异步任务 Worker
   (向量生成/索引重建/日志统计)
```

### 8.3 架构说明

* 前端负责展示与交互；
* Nginx 负责静态托管与 API 转发；
* FastAPI 负责所有业务逻辑；
* PostgreSQL 存放订单、用户、商品、会员、评价等关系型数据；
* pgvector 存放商品向量、文化内容向量、用户偏好向量；
* Redis 负责缓存热门商品、首页推荐、验证码、登录状态黑名单、异步任务协调；
* MinIO 存储商品图片、文化图片、评价附图等；
* Worker 负责批量 embedding、向量重建、用户画像更新、推荐日志聚合。

## 9.7 智能搜索与推荐模块

这是本项目的核心亮点模块。

### 9.7.1 模块目标

让用户可以用自然语言表达需求，并获得更符合语义与文化偏好的结果，而不是只能依赖精确关键词。

### 9.7.2 支持的搜索方式

1. **关键词搜索**

   * 如“香囊”“折扇”“宋制汉服”
2. **筛选搜索**

   * 类别、价格、节令、工艺、会员价、热度
3. **自然语言搜索**

   * 如“适合送老师的雅致国风礼物”
   * 如“颜色素一点、适合春天拍照的明制汉服”
4. **相似商品搜索**

   * 浏览商品详情页后召回风格相近商品
5. **文化联动搜索**

   * 搜“端午习俗”时，不只返回文章，也返回相关香囊、龙舟文创等商品

### 9.7.3 推荐入口设计

* 首页“猜你喜欢”
* 商品详情页“相关商品”
* 商品详情页“相关文化”
* 分类页“为你推荐”
* 节令专题“应景推荐”
* 会员中心“会员专属推荐”

### 9.7.4 推荐策略设计

本项目推荐系统建议采用**混合推荐架构**：

#### A. 规则召回

* 热门商品
* 新品商品
* 节令商品
* 会员专属商品
* 活动商品

#### B. 向量召回

基于以下文本向量化：

* 商品标题
* 商品描述
* 商品文化简介
* 标签与场景描述
* 文化专题内容
* 用户最近行为摘要

#### C. 行为偏好召回

根据用户：

* 浏览记录
* 搜索记录
* 加购记录
* 下单记录
* 收藏记录（若实现）
* 评价关键词

构建用户偏好标签，如：

* 偏爱“明制”
* 偏爱“素雅配色”
* 偏爱“节令礼品”
* 偏爱“非遗手工艺”
* 偏爱“300 元以内送礼场景”

#### D. 混合排序

最终得分可按如下思路设计：

```text
final_score =
0.35 * semantic_similarity
+ 0.20 * user_preference_match
+ 0.15 * category_interest_match
+ 0.10 * popularity_score
+ 0.10 * seasonal_score
+ 0.05 * member_benefit_score
+ 0.05 * freshness_score
```

### 9.7.5 为什么这个方案适合比赛

因为它具备三个优势：

1. **容易讲清楚**：评委能理解“规则 + 语义 + 行为”的混合逻辑；
2. **容易落地**：不依赖超复杂训练流程；
3. **容易演示效果**：可以现场输入一句自然语言，看结果明显优于关键词搜索。

## 10. 向量数据库设计

## 10.1 为什么要使用向量数据库

使用向量数据库的原因不是为了“追热点”，而是因为本项目的检索对象天然适合语义表达：

* 用户搜索语句往往很口语化
* 商品与文化内容都是长文本
* 文化偏好难以用单一标签表达
* 商品相似性存在“风格相似”“意境相似”“适用场景相似”而不只是名称相似

### 10.2 向量化对象

建议对以下内容生成向量：

1. 商品文本向量
   标题 + 副标题 + 描述 + 标签 + 文化背景 + 场景

2. 文化内容向量
   文章标题 + 正文分块 + 关键词 + 标签

3. 用户兴趣向量
   最近浏览商品文本聚合 + 搜索词 + 历史偏好标签

### 10.3 检索流程

#### 自然语言搜索流程

```text
用户输入查询
 -> 查询预处理
 -> 文本向量化
 -> pgvector 相似度召回 TopK
 -> 结合类目/价格/节令过滤
 -> 业务排序
 -> 返回结果
```

#### 商品详情推荐流程

```text
当前商品
 -> 读取商品向量
 -> 在商品向量库中检索相似商品
 -> 排除本商品与缺货商品
 -> 引入用户偏好加权
 -> 返回相关推荐
```

### 10.4 索引策略

* 商品向量表：按上架商品建立索引
* 文化内容分块表：按 chunk 建立索引
* 定时重建或增量更新向量
* 新增/修改商品时异步生成 embedding
* 删除商品时同步删除索引记录

## 20. 创新点总结

本项目的创新点主要体现在以下四个方面：

### 20.1 文化电商融合

将传统文化传播与电商系统结合，不只是卖商品，还讲清商品背后的文化来源、工艺价值与历史背景。

### 20.2 向量语义检索

引入向量数据库，使用户可以用自然语言搜索商品，提升传统文化商品的发现效率。

### 20.3 可解释混合推荐

不是简单调用大模型，而是通过“规则召回 + 向量召回 + 行为偏好排序”的方式，形成可解释、可落地的推荐系统。

### 20.4 适合比赛展示的完整闭环

从用户到订单，从文化内容到推荐，从前台到后台，从部署到答辩，形成完整可演示系统。

## 22. 最终结论

“拾遗阁”不是一个普通商城，而是一个面向中华优秀传统文化消费场景的**智能文化电商 Web 平台**。项目以电商业务为基础，以文化内容为灵魂，以向量数据库为技术亮点，以推荐系统为创新抓手，兼顾：

* 实用性
* 完整性
* 创新性
* 可部署性
* 可答辩性

从比赛角度看，这个项目最重要的不是“做得多复杂”，而是“做得完整、做得稳定、做得能讲清楚”。
因此本项目最优实现路线应为：

**先完成后端业务闭环，再完成内容体系，最后完成向量检索和推荐增强。**
